# Plan: DirectInput Button Names + Press-to-Assign

## Problem
Button selection uses hard-coded ComboBox items ("Button 1"..."Button 32") with no relation to the actual device. Users must know which generic button number maps to their physical button. Additionally, button selections are not persisted across restarts.

## Solution: Two Improvements

### A. Pull Real Button Names from DirectInput

SharpDX.DirectInput's `Joystick.GetObjects()` returns `DeviceObjectInstance` items for every device object (buttons, axes, POVs). Each has a `Name` property containing the driver-reported label (e.g. "Button 1", "Trigger", "Thumb Left").

**FfbDeviceManager.cs changes:**

1. Add `GetButtonNames()` method that enumerates `Joystick.GetObjects()`, filters for `ObjectTypeGuid == DeviceObjectTypeGuid.Button` (or checks `ObjectInstance.Offset` in the button range), and returns a `List<string>` with one name per button index.

```csharp
public List<string> GetButtonNames()
{
    if (_device == null) return new();
    var names = new List<string>();
    foreach (var obj in _device.GetObjects())
    {
        if (obj.ObjectType == SharpDX.DirectInput.ObjectTypeGuid.Button)
        {
            int idx = obj.ObjectId.RawName; // or use offset to compute index
            // pad list to fit index
            while (names.Count <= idx) names.Add($"Button {names.Count + 1}");
            names[idx] = !string.IsNullOrWhiteSpace(obj.Name) ? obj.Name : $"Button {idx + 1}";
        }
    }
    return names;
}
```

2. Add `GetSecondaryButtonNames()` — same logic but for `_secondaryDevice`.

3. Expose as properties: `PrimaryButtonNames` and `SecondaryButtonNames`.

**MainViewModel.cs changes:**

4. Add `ObservableCollection<string> SnapshotButtonNames { get; }` and `ObservableCollection<string> PanicButtonNames { get; }`.
5. When primary device connects → populate `SnapshotButtonNames` from `FfbDeviceManager.GetButtonNames()`. Always prepend "Disabled" as index 0.
6. When secondary device connects → populate `PanicButtonNames` from `FfbDeviceManager.GetSecondaryButtonNames()`. Always prepend "Disabled".

**MainWindow.xaml changes:**

7. Replace hard-coded `<ComboBoxItem>` lists with `ItemsSource="{Binding SnapshotButtonNames}"` and `ItemsSource="{Binding PanicButtonNames}"`. Keep `SelectedIndex` binding as-is.

### B. Add "Press to Assign" Mode (Optional but Highly Recommended)

This is the gold-standard UX for sim racing tools. Instead of picking from a dropdown, the user clicks "Assign", then presses any button on the device.

**FfbDeviceManager.cs:**

8. Add `event Action<int, string>? ButtonPressedForAssignment` that fires during polling when a rising edge is detected in assignment mode.

**MainViewModel.cs:**

9. Add `bool IsAssigningSnapshotButton` and `IsAssigningPanicButton` flags.
10. Add `RelayCommand AssignSnapshotButtonCommand` / `AssignPanicButtonCommand` that set the flag and change status text to "Press a button on your wheel...".
11. In `PollSnapshotButton()` / `PollPanicButton()`, when the assign flag is set, detect any rising edge → set the combo index, clear the flag, update status.

**MainWindow.xaml:**

12. Add a small "Assign" button next to each button ComboBox. When clicked, enters listen mode (highlight the ComboBox or show "Listening..." text).
13. The "Pressed:" feedback line already exists for snapshot — extend to show during assignment too.

### C. Persist Button Selections (Bug Fix)

**AppSettings.cs:**

14. Add properties: `SnapshotButtonComboIndex`, `PanicButtonComboIndex`, `SelectedPanicDeviceInstanceId` (string - GUID).

**MainViewModel.cs:**

15. Save these on change. Load on startup and re-apply after device enumeration completes.

7. *(Optional phase 2)* — Add press-to-assign mode

## Risk / Open Questions

- **Do racing wheels report useful button names?** Many DirectInput devices just report "Button 1", "Button 2", etc. Moza, Fanatec, Logitech wheels may or may not provide friendly names. The press-to-assign feature (part B) is the real UX win regardless of name quality.
- **Button object enumeration offset mapping** — SharpDX `DeviceObjectInstance.ObjectId` may not directly map to the button array index. We need to verify the mapping (likely via `ObjectId.InstanceNumber` or the offset within the data format). This needs testing with the actual wheel.

## Recommendation

Implement parts A + B + C together. Part A alone may not provide much value if wheels only report generic names, but combined with press-to-assign (B) and persistence (C), the UX improvement is significant.
