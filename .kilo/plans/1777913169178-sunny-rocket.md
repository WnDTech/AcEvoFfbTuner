# HF8 / HF8 Pro Haptic Pad Support Investigation

## Device Summary

**Next Level Racing HF8** (NLR-G001, ~$179) and **HF8 Pro** (NLR-G006, ~$299) are wearable haptic feedback pads with 8 individually controllable vibration motors, designed to strap onto sim racing/flight seats.

| Feature | HF8 | HF8 Pro |
|---------|-----|---------|
| Motors | 8 zones | 8 zones (200% more torque, 400% more power, 60% faster RPM) |
| PC Connection | USB HID | USB HID |
| Console | 3.5mm audio jack | 3.5mm audio jack |
| Software | HFS (15fps) or SimHub | HFS (75fps) or SimHub |
| SKU | NLR-G001 | NLR-G006 |

## Current App Architecture (relevant components)

```
TelemetryLoop
  └─ FfbPipeline
       ├─ FfbVibrationMixer  → kerb/slip/road/ABS/scrub/rear-slip signals
       ├─ FfbLfeGenerator    → low-frequency suspension+RPM envelope
       ├─ FfbChannelMixer    → final force mixing
       └─ FfbOutputClipper   → final clipping
  └─ FfbDeviceManager
       ├─ DirectInput constant force → wheel
       ├─ DirectInput periodic vibration → wheel
       └─ WheelLedController → HID output to wheel LEDs (Fanatec/Moza/Logitech/Simucube)
```

The app already has rich vibration/haptic signal generation in `FfbVibrationMixer` and `FfbLfeGenerator` that is currently mixed into the wheel's periodic effect. These signals could be split out and sent to the HF8's 8 independent motors.

## Integration Challenges

### 1. Proprietary HID Protocol (CRITICAL BLOCKER)
- NLR does **not publish a public SDK or API** for third-party direct control of the HF8 motors.
- The HFS software communicates with the device via USB HID output reports.
- SimHub has HF8 integration, meaning the protocol has been reverse-engineered by at least one third party.
- **Without the HID report format, we cannot send motor commands directly.**

### 2. VID/PID Unknown
- The HF8's USB Vendor ID and Product ID are not documented publicly.
- Would need to be discovered by plugging in a device and checking Windows Device Manager or using `SetupDi` enumeration (our `WheelLedController` already has this infrastructure).

### 3. Conflict with HFS Software
- If the user runs HFS software, it will hold the HID device handle exclusively.
- Our app already handles this pattern (e.g., Moza Pit House conflicts with serial/HID LED control).
- The app currently lists SimHub as a conflicting process to close.

## Proposed Integration Approach

### Phase 1: Protocol Discovery (requires HF8 hardware)
1. Connect HF8/HF8 Pro to PC via USB
2. Use Windows Device Manager or our existing `SetupDi` HID enumeration to discover:
   - VID/PID of the device
   - HID usage page/usage
   - Output report byte length
   - Feature report byte length
3. Capture HID traffic using a USB sniffer (e.g., Wireshark + USBPcap) while the HFS software controls the device
4. Analyze the output report format to determine the motor command protocol
5. Document the protocol: byte layout for 8 motor intensities, any header/checksum bytes

### Phase 2: Hf8HapticController Implementation
Create `src/AcEvoFfbTuner.Core/DirectInput/Hf8HapticController.cs`:

```
class Hf8HapticController : IDisposable
{
    // 8 motor zones: FL/FR/RL/RR tire, seat left/right, back upper/lower
    float[] MotorIntensities = new float[8];  // 0.0 - 1.0 per motor
    
    bool TryConnect();          // Scan HID devices, find HF8 by VID/PID
    void SendMotorUpdate();     // Send HID output report with 8 motor values
    void Disconnect();
    void Dispose();
}
```

Motor zone mapping from our telemetry signals:
| Motor Zone | Signal Source |
|-----------|---------------|
| Front Left (0) | FL suspension delta + FL slip ratio |
| Front Right (1) | FR suspension delta + FR slip ratio |
| Rear Left (2) | RL suspension delta + RL slip ratio |
| Rear Right (3) | RR suspension delta + RR slip ratio |
| Seat Left (4) | LFE output + lateral G |
| Seat Right (5) | LFE output + lateral G |
| Back Upper (6) | RPM envelope + gear shift pulse |
| Back Lower (7) | Road vibration + ABS pulse |

### Phase 3: Integration into TelemetryLoop
1. Add `Hf8HapticController` as an optional output alongside `FfbDeviceManager`
2. After `FfbPipeline.Process()`, extract vibration/LFE/suspension signals
3. Map signals to 8 motor zones and send HID output report at ~75fps (HF8 Pro) or ~15fps (HF8)
4. Add enable/disable toggle and per-zone gain controls to the profile

### Phase 4: UI
- Add "Haptic Pad" section to the Settings/Profile UI
- Per-motor-zone gain sliders
- Enable/disable toggle
- Connection status indicator

### Phase 5: SimHub Coexistence Path (Alternative)
If direct HID protocol proves too fragile, provide a lightweight alternative:
- Output haptic data to a named pipe or UDP port that SimHub can consume
- This avoids protocol reverse-engineering but requires SimHub to be running
- Our app already has telemetry data; we'd just need to format it for SimHub's input plugins

## Decision: Direct HID Control (user has device)

User owns an HF8/HF8 Pro and wants standalone direct HID control without SimHub dependency.

## Implementation Plan

### Step 1: Build HID Discovery Tool
Create a small diagnostic tool that reuses our existing `SetupDi`/HID enumeration code from `WheelLedController.cs` to:
- List all connected HID devices with VID, PID, usage page, usage, output report length, feature report length, and product string
- Write results to a log file at `AppData/Roaming/AcEvoFfbTuner/hf8_discovery.log`
- The user runs this with the HF8 connected so we can identify the device's VID/PID and HID characteristics

### Step 2: Protocol Capture
With the device identified:
1. Start the HFS software and enable various effects (RPM, braking, gear shift, road bumps)
2. Use Wireshark + USBPcap to capture USB traffic to the HF8
3. Analyze output report byte patterns:
   - Identify the report ID byte
   - Map byte positions to motor zones (send one motor at a time via HFS and observe which bytes change)
   - Determine value range (0-255? 0-65535? signed/unsigned?)
   - Check for any header, checksum, or framing bytes
4. Document the protocol

### Step 3: Implement Hf8HapticController
- Create `src/AcEvoFfbTuner.Core/DirectInput/Hf8HapticController.cs`
- Reuse HID enumeration patterns from `WheelLedController.cs`
- Implement: `TryConnect()`, `SendMotorUpdate(float[] 8 intensities)`, `Disconnect()`
- Add to `FfbDeviceManager` as a parallel output device (alongside wheel FFB and LEDs)

### Step 4: Signal Mapping & Integration
Wire into `TelemetryLoop` after `FfbPipeline.Process()`:
- Extract per-wheel signals: `slipRatio[4]`, `suspensionTravel[4]`, `slipAngle[4]`
- Extract global signals: `FfbVibrationMixer` outputs (road, kerb, ABS, scrub, rear slip), `FfbLfeGenerator.LfeOutput`
- Map to 8 HF8 motor zones with configurable gains
- Send at appropriate rate (75fps for HF8 Pro, 15fps for original HF8)

### Step 5: Profile & UI
- Add HF8 settings to `FfbProfile` (enable, per-zone gains, master gain)
- Add UI section in MainWindow for HF8 configuration
- Show connection status

### Step 6: Build, Test, Tune
- Build with `dotnet build AcEvoFfbTuner.slnx -c Release`
- User tests with AC EVO driving
- Tune motor mapping based on feel
