# Plan: Oscillation Guard — Replace LiveAutoTuner with Focused Oscillation Suppression

## Problem
Testers can't drive a lap because their wheels oscillate violently. The current `LiveAutoTuner`:
- Runs at **UI tick rate** (~30Hz via `OnUiUpdate`) — too slow to catch oscillation
- Makes **tiny corrections** (3-7%) with **10-30 second cooldowns** — too conservative
- Tries to fix everything (clipping, center oscillation, jitter, road vibration, etc.) — unfocused
- Modifies pipeline parameters permanently — can make things worse before better

## Solution: Real-Time Oscillation Guard

### Core Concept
A **post-processing filter** that runs **inside the FFB pipeline** at ~333Hz tick rate. It detects oscillation from force output patterns and **immediately** applies a stability multiplier to suppress it. No parameter modification, no cooldowns, no guessing — pure signal-level filtering.

### Architecture

```
Physics → ChannelMixer → LUT → Slip → Damping → Dynamic → TyreFlex → Clipper
  → Slew Rate → [OSCILLATION GUARD] → Equalizer → Vibrations → Device
```

The guard sits after slew rate limiting (the last "safety" stage) and before the equalizer/vibrations (content stages). This way it can clamp the force signal without affecting vibration effects.

## Files to Create/Modify

### 1. NEW: `src/AcEvoFfbTuner.Core/FfbProcessing/OscillationGuard.cs`

**Detection** (runs every frame at ~333Hz):
- Circular buffer of last 32 output force values
- Count zero-crossings (sign changes) in the buffer
- Track peak-to-peak amplitude
- **Oscillation = >8 zero-crossings in 32 frames AND peak-to-peak > 0.15**

**Suppression**:
- `StabilityFactor`: 1.0 (no suppression) → 0.3 (heavy suppression)
- When oscillation detected: factor drops **fast** (converges in ~5 frames to 0.3-0.5)
- When oscillation stops: factor **slowly recovers** (0.01 per frame, takes ~70 frames = 200ms to fully recover)
- Output = `force * StabilityFactor`

**Force direction detection** (bonus diagnostic):
- Track correlation between force direction and steer velocity
- If force consistently pushes same direction as wheel movement (positive feedback) → suggest force inversion
- Exposed as `ForceDirectionWarning` flag

**Public API**:
```csharp
public float Apply(float force, float steerAngle)
public bool IsOscillating { get; }
public float StabilityFactor { get; }      // 0.0-1.0
public float OscillationLevel { get; }     // 0.0-1.0 (how bad)
public bool ForceDirectionWarning { get; }  // suggests ForceInvertEnabled
public void Reset()
```

## Key Design Decisions

### Why a signal filter, not parameter modification?
- **Instant**: Works within 5 frames (~15ms), not 10-30 seconds
- **Non-destructive**: Doesn't permanently alter the user's profile settings
- **Self-recovering**: Automatically relaxes when oscillation stops
- **Predictable**: Simple multiplication, easy to understand and debug

### Why after slew rate, before equalizer?
- Slew rate limiting is the first line of defense
- OscillationGuard is the second line — catches what slew rate can't
- Equalizer and vibrations are content, not safety — guard shouldn't affect them

### Why not just increase damping?
- Damping is velocity-dependent and requires careful tuning per wheelbase
- The guard works regardless of damping settings
- It's a blunt instrument: when the wheel is going crazy, just turn down the gain until it stops

### What about the force inversion detection?
- A common cause of oscillation is wrong force direction (positive feedback loop)
- The guard can detect this pattern: force consistently correlates with steer velocity
- Shows a clear warning to the user instead of silently trying to compensate
- This addresses the Moza vs Simagic sign issue that's been plaguing testers
