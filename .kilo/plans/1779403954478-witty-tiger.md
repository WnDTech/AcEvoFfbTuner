# Raceroom Racing Experience Shared Memory Integration Plan

## Goal
Add Raceroom Racing Experience (R3E) shared memory integration to ACEVO - Telemetry FFB while keeping AC EVO integration completely separate. Provide a toggle in the app to switch between the two games.

## Key Principles
1. **Zero Touch Existing EVO Code**: Do not modify any existing AC EVO shared memory code
2. **Separate Implementation**: Create parallel systems for R3E that mirror EVO systems
3. **Game Selection Toggle**: Add UI control to switch between AC EVO and R3E
4. **Shared Processing Pipeline**: Both games feed into the same FFB processing pipeline after data mapping

## Architecture Overview

### 1. Shared Memory Reader Abstraction
Create an interface and implementations for both games:
- `ISharedMemoryReader` interface with methods: `TryConnect()`, `Disconnect()`, `TryReadPhysics()`, `TryReadGraphics()`, `TryReadStatic()`
- `AcEvoSharedMemoryReader` (existing, unchanged)
- `RaceroomSharedMemoryReader` (new implementation)

### 2. Data Structure Mapping
Create R3E-specific structs that mirror the EVO structs but map to R3E shared memory layout:
- `SPageFilePhysicsR3e`
- `SPageFileGraphicR3e` 
- `SPageFileStaticR3e`

### 3. Telemetry Loop Adaptation
Modify `TelemetryLoop` to work with the abstraction:
- Accept `ISharedMemoryReader` instead of hardcoded `SharedMemoryReader`
- Maintain separate latest data stores for each game type
- Keep all processing logic identical

### 4. Game Selection Mechanism
Add enum and state management:
```csharp
public enum SupportedGame { AcEvo, Raceroom }
```
- Add setting to MainViewModel for selected game
- Initialize appropriate SharedMemoryReader based on selection
- Update connection/disconnection logic to use selected reader

### 5. UI Changes
- Add game selector in Settings page or header
- Update connection status messages to show active game
- Preserve all existing UI functionality

## Detailed Implementation Steps

### Phase 1: Foundation (No UI Changes)
1. Create `ISharedMemoryReader` interface in `SharedMemory` folder
2. Create `RaceroomSharedMemoryReader` class implementing the interface
3. Define R3E structs matching the memory layout from research
4. Modify `TelemetryLoop` constructor to accept `ISharedMemoryReader`
5. Update `MainViewModel` to instantiate and manage the reader

### Phase 2: Game Selection
1. Add `SupportedGame` enum
2. Add `SelectedGame` property to MainViewModel with persistence
3. Add UI toggle (dropdown or toggle switch) in Settings page
4. Implement game switching logic that reinitializes the reader

### Phase 3: Testing & Validation
1. Verify AC EVO still works unchanged
2. Test R3E connection and data reading
3. Validate FFB output works with both games
4. Ensure snapshot and recording features work with both

## Risks and Mitigations

### Risk: Memory Layout Differences
- Mitigation: Thoroughly research R3E shared memory layout from r3e-api
- Mitigation: Implement extensive logging/debugging for R3E data validation

### Risk: Threading Issues
- Mitigation: Keep threading model identical to existing implementation
- Mitigation: Use same locking patterns as TelemetryLoop

### Risk: Configuration Persistence
- Mitigation: Use existing AppSettings mechanism for game selection
- Mitigation: Ensure selection survives app restarts
